텍스트와 폭

"한 글자"는 생각보다 안 단순하다 — code point · grapheme · 왜 한글/이모지는 두 칸일까

이 레슨의 win 격자 한 칸(셀)에 들어가는 "글자"가 바이트도 code point 도 아닌 grapheme이라는 것, 그리고 한글·한자·이모지가 왜 두 칸(폭 2)을 먹는지를 이해한다. 이게 리뷰에서 커서·선택·폭 버그의 뿌리다.
이어서 기초 ②에서 "글자 바이트를 격자에 찍는다"고 했다. 그런데 "글자 하나"가 정확히 뭐냐가 이번 주제다 — 여기서 셀의 width 필드와 나중에 배울 "wide-char spacer"가 나온다.

1. 바이트 → code point (UTF-8)

화면에 찍을 글자는 UTF-8 바이트로 온다. 문제는 한 글자가 여러 바이트일 수 있다는 것 — 영어 a는 1바이트지만 한글 가는 3바이트다:

a → 0x61 (1 byte → code point U+0061)
가 → 0xEA 0xB0 0x80 (3 bytes → code point U+AC00)

그래서 엔진은 바이트를 모아서 code point 로 디코드한 뒤에야 "무슨 글자"인지 안다. (code point = 유니코드가 글자마다 매긴 번호, U+AC00 같은 것.)

2. code point ≠ 사람이 보는 글자 (grapheme)

여기가 함정. 사람이 "한 글자"로 보는 게 code point 여러 개일 수 있다:

é
e(U+0065) + ́(U+0301, 악센트) = 2 code point, 1 글자
👍🏽
손 이모지 + 피부색 조절자 = 2 code point, 1 글자
👨‍👩‍👧
사람 셋 + 결합기호(ZWJ) = 여러 code point, 1 글자

이렇게 사람이 한 글자로 인식하는 code point 묶음을 grapheme cluster(줄여서 grapheme)라 부른다(유니코드 규칙 UAX #29). 셀 한 칸엔 grapheme 이 들어간다 — code point 가 아니라. 그래서 é 는 두 code point 여도 한 칸에 들어간다.

justerm 연결 그래서 justerm 의 셀은 content 를 grapheme 으로 담는다(흔한 건 인라인, 드문 다중 code point 는 side-table 참조 — 셀 크기 고정 유지). 용어집의 "Cell", "Grapheme" 항목이 이거.

3. 폭 — 한 칸이냐 두 칸이냐 (East Asian Width)

격자는 등폭(monospace)이라 칸이 다 같은 너비다. 그런데 한글·한자·많은 이모지는 두 칸을 차지한다. 유니코드가 글자마다 "이건 Wide(넓음)"라고 표시해둔 규칙이 East Asian Width(UAX #11)다.

예: a가b 를 격자에 놓으면 —

a가b

가 가 두 칸(파란 칸)을 먹어서 총 4칸이 아니라 1+2+1 = 4칸… 이 아니라 a(1) + 가(2) + b(1) = 4칸을 쓴다. 그래서 셀에는 width 필드(1 또는 2)가 있다.

숨은 함정 — wide-char spacer 두 칸짜리 글자를 격자에 넣으면, 실제 글자는 왼쪽 칸에 있고 오른쪽 칸은 "비었지만 빈 게 아닌" 자리표시(spacer)가 된다:
a가·b
이 뒷칸(spacer)을 그냥 빈칸으로 취급하면 커서·선택·덮어쓰기가 다 어긋난다. justerm 이 추적하는 "wide-char spacer" 숨은 상태가 이거 — 지금은 "이런 게 있다"만 알면 충분(트랙 2에서 다룸).

4. combining mark = 0칸

반대로 §2 의 악센트 ́ 같은 결합문자는 0칸이다 — 새 칸을 만들지 않고 앞 글자에 얹힌다. 커서도 안 움직인다. 그래서 é(e + 악센트)는 두 code point 여도 격자에선 한 칸, 커서는 한 번만 전진.

5. 왜 리뷰에서 중요한가

폭 하나를 틀리면 도미노다 — 커서 위치, 선택 범위, 덮어쓰기, 줄바꿈이 전부 어긋난다. 그래서 "이 글자가 몇 칸이냐"(width)는 core 의 책임이고, 렌더러가 아니라 엔진이 정확히 알아야 한다.

우리가 이미 본 실사례 검수 ①의 #297 사례 — 이모지 폭 계산 갭을 렌더러가 몰래 덮은(우회) 그거 — 가 바로 이 영역(East Asian Width / 폭)이다. 폭은 core 책임인데 렌더러가 손대서 문제였던 것. 이제 그 사례가 왜 폭 얘기였는지 이해될 거다.
한 줄 요약 바이트 → (UTF-8) → code point → (묶으면) 사람이 보는 grapheme = 셀 한 칸의 내용. 한글·한자·이모지는 폭 2(East Asian Width)라 두 칸 + 뒷칸은 spacer. 결합문자는 폭 0. 폭을 틀리면 커서·선택이 다 무너지니 폭은 core 책임.
