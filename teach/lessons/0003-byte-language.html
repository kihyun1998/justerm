바이트의 언어

이스케이프 시퀀스 · CSI · SGR · OSC — 그리고 파서가 왜 상태머신인가

이 레슨의 win ESC[31m 같은 명령을 보면 "아, 이건 색 바꾸는 CSI/SGR 명령이구나"를 읽을 수 있게 된다. 그리고 파서(justerm 이 vte 에 맡기는 그 부분)가 왜 상태머신인지 이해한다.
이어서 기초 ①에서 "바이트 = 글자 + 명령"이라 했다. 글자 절반은 이미 이해했다. 이번엔 그 명령 절반을 뜯어본다.

1. ESC 가 신호탄이다

바이트 하나 0x1b (ESC) 가 "지금부터 오는 건 글자가 아니라 명령이야"의 신호다. 글로 쓸 땐 \x1b 또는 ESC 로 표기한다(눈에 안 보이는 바이트라서). ESC 뒤에 따라오는 몇 바이트가 "어떤 명령인지"를 정한다.

참고 — 명령이 다 ESC 로 시작하진 않는다 아주 기본적인 것들은 ESC 없이 바이트 하나로 동작한다(C0 컨트롤): 0x0a=줄바꿈(LF), 0x0d=맨 앞으로(CR), 0x09=탭(HT), 0x08=백스페이스(BS), 0x07=삑소리(BEL). 나머지 풍부한 명령이 ESC 로 시작한다.

2. 명령의 세 가문

ESC 로 시작하는 명령은 다음 글자로 가문이 갈린다:

ESC [ …
CSI (Control Sequence Introducer) — 제일 큰 가문. 커서 이동·화면 지우기·스크롤·색. 거의 다 여기.
ESC ] …
OSC (Operating System Command) — 문자열을 나르는 명령. 창 제목·클립보드·현재 폴더 등.
ESC (그 외)
짧은 단발 명령 (예: ESC c = 터미널 리셋).

그러니까 ESC[ 를 보면 → "CSI 명령이구나", ESC] 를 보면 → "OSC(문자열)구나" 로 읽으면 된다.

3. CSI 한 개는 어떻게 생겼나

CSI 명령의 뼈대는 항상 이렇다:

ESC [ 파라미터 ; 파라미터 최종문자
  • 파라미터 = 숫자들, ; 로 구분 (예: 1;31)
  • 최종문자 = 이 명령이 무엇인지 정하는 알파벳 한 글자 (예: m=색/스타일, H=커서이동, J=화면지우기)

예를 들어:

ESC [ 2 J → 화면 전체 지워라(J), 방식 2(전체)
ESC [ 10 ; 5 H → 커서를 10행 5열로(H)

4. SGR — 색과 스타일 (CSI 중 최종문자 m)

네가 제일 자주 보게 될 CSI 는 최종문자가 m 인 것 — 이걸 SGR (Select Graphic Rendition)이라 부른다. 파라미터가 "무슨 스타일"인지 정한다:

ESC[31mworldESC[0m → 빨강 켜고 "world" 찍고, 리셋
0
리셋 (스타일 다 끔)
1
굵게(bold)
3 / 4
기울임 / 밑줄
30–37
글자색 (31=빨강, 32=초록 …)
40–47
배경색

ESC[1;31m 처럼 여러 개를 ; 로 한꺼번에 준다(굵게 + 빨강). 여기서 색이 31 같은 번호로 온다는 데 주목 — 실제 hex 가 아니다. 이게 나중에 "color reference"(유닛 4)로 이어진다: 엔진은 "31번 색"만 기억하고, 그게 어떤 빨강인지는 렌더러가 정한다.

5. OSC — 문자열 명령 (곁가지)

ESC ] 로 시작하는 OSC 는 숫자 하나 + ; + 문자열 + 끝표시(BEL)로 온다:

ESC ] 0 ; My Terminal BEL → 창 제목을 "My Terminal"로

0/2=창 제목, 7=현재 폴더, 8=하이퍼링크. 지금은 "이런 가문이 있다"만 알면 충분.

CSI 와 OSC 는 "이름 위치"가 정반대다
CSI: ESC [ 3 1 m ← 최종문자 m = "무슨 명령" + 여기서 끝
OSC: ESC ] 0 ; My Terminal BEL ← 앞 숫자 0 = "무슨 명령" / BEL = 그냥 "문자열 끝"
  • CSI: "무슨 명령"이 끝(최종문자)에 있다 — 끝문자가 이름 겸 종료.
  • OSC: "무슨 명령"이 앞(숫자)에 있고, 끝의 BEL 은 이름이 아니라 "문자열 여기까지"라는 마침표일 뿐.
왜 다르냐: OSC 는 창제목·URL 같은 길이도 내용도 제멋대로인 자유 문자열을 나른다 → 명령 종류를 앞에 둬야 하고(길이를 모르니 먼저 읽어야), 끝을 명시해야 한다. CSI 는 짧은 숫자만 나르니 끝 알파벳 하나로 "끝 + 이름"을 동시에 해결. (끝표시는 BEL 0x07 또는 ST = ESC \, 둘 다 "문자열 끝" 뜻.)

6. 파서는 왜 상태머신인가

이 모든 걸 바이트 흐름에서 알아내는 게 파서다. 그냥 "ESC[ 가 보이면…" 식으로 문자열 매칭하면 안 되는 이유가 둘 있다:

  • 바이트가 아무렇게나 쪼개져 온다. ESC[31m 이 ESC[3 까지 오고 나머지는 다음 번에 올 수 있다(네트워크·버퍼).
  • 이상한 입력도 온다. 깨진 시퀀스가 와도 파서가 꼬이거나 멈추면 안 된다.

그래서 파서는 "지금 어느 상태냐"를 들고 바이트를 하나씩 먹으며 상태를 옮긴다. 아주 단순화하면:

ground
글자 찍는 중 — ESC 봄 → escape
가문 기다림 — [ 봄 → csi
파라미터 모음 — 최종문자 → 실행
→ ground 복귀

ESC[31m 이 오면: ESC(→escape) → [(→csi) → 3,1(파라미터 쌓기) → m(최종문자 = "SGR 실행!" 하고 ground 복귀). 쪼개져 와도 상태가 남아있으니 안 꼬인다.

justerm 연결 이 상태머신은 Paul Williams 의 표준 설계(vt100.net)이고, justerm 은 이 진짜 어려운 파싱만 검증된 크레이트 vte 에 맡긴다. 그 위에 격자·스크롤백을 얹는 게 justerm-core. → 리뷰에서 "이 시퀀스가 뭐 하는 거지?"는 파서 코드가 아니라 xterm ctlseqs 사전에서 찾는다.
한 줄 요약 ESC = 명령 신호탄. ESC[=CSI(대부분), ESC]=OSC(문자열). CSI 는 파라미터 + 최종문자 구조고, 최종문자 m이면 색/스타일(SGR). 색은 번호로 온다. 파서는 쪼개진·깨진 입력에도 안 꼬이려고 상태머신 — 그 부분을 justerm 은 vte 에 위임.
