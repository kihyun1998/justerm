justerm 은 하나가 아니다

패밀리, 그리고 크레이트마다 다른 경계 — 리뷰의 첫 질문은 "어느 크레이트냐"

먼저 이것부터 이 레슨은 damage · wire · 셀 · DecodedFrame 같은 말을 쓴다. 낯설면 기초 ① 터미널이란(그리드·셀·커서·스크롤백·VT스트림)과 기초 ②(damage·wire)를 먼저 보고 오면 훨씬 편하다.
이 레슨의 win 에이전트가 짠 변경을 보면 ①"어느 크레이트냐" → ②"그 크레이트의 경계를 넘었나" 두 질문으로 리뷰할 수 있게 된다. 터미널 프로젝트에서 가장 흔한 실수 한 종류를 통째로 잡는 체크리스트.

1. 흔한 한 줄부터 버리자

많은 사람이(그리고 이 repo 의 옛 문서들도) 이렇게 말했다:

절반만 맞는 말 "justerm 은 순수 터미널 엔진이다. 그림은 안 그린다."

이건 절반만 맞다. 그리고 그 절반을 전부로 착각하면 리뷰가 통째로 틀어진다. 진실은:

justerm 은 크레이트 하나가 아니라 패밀리(우산 이름)다. 그 안의 justerm-core 가 "순수 엔진"이고 — 패밀리 전체는 xterm 처럼 엔진 + 렌더러 + 웹 위젯을 다 갖춘 풀스택 터미널이다.

그래서 "justerm 은 안 그린다"는 문장은 justerm-core 는 안 그린다로 읽어야 맞다. 렌더러(justerm-renderer)는 그리는 게 임무다. 이 구분을 놓치면 멀쩡한 렌더러 코드를 "경계 위반"이라고 반려하게 된다(§5 에서 실제로 겪어본다).

2. 패밀리 지도 — 5개 크레이트

각자 역할이 다르고, 경계도 다르다. 리뷰 전에 이 지도를 머리에 넣어둔다.

justerm-core — 엔진

  • 바이트 → 상태(그리드·스크롤백·커서·selection)
  • damage 프레임을 wire 포맷으로 encode
  • 안 그림 · 안 읽음 · 안 보냄 · 색은 참조만

justerm-wasm-decode — 디코더

  • core 의 decode 를 WASM 으로 컴파일
  • 네이티브·웹이 디코더 하나를 공유(TS 재작성 X)
  • wire 바이트 → DecodedFrame(열 배열)

justerm-renderer — 렌더러

  • DecodedFrame + 주입된 팔레트 → WebGL2 로 그림
  • 그리는 게 임무! (단 팔레트 주입 = theme-agnostic 유지)
  • beamterm(외부) 대체하는 자체(first-party) 렌더러

justerm-web — 위젯

  • 브라우저 터미널 위젯(웹 개발자가 갖다 씀)
  • DecodedFrame 받아 렌더러에 넘겨 조율
  • 프레임 모드(지금) / in-wasm(나중), FrameSource seam 뒤

다섯째 justerm-facade 는 옛 이름(justerm 0.5.1) 묘비 — 개명 전 이름 쓰던 사람 안 깨지게 재수출만 하는 죽은 크레이트다. 리뷰에서 볼 일 거의 없다.

빨간 두 상자가 justerm 이 직접 소유하는 Rust 층(엔진 + 렌더러). IPC(전송)는 크레이트가 아니라 소비처(PenTerm 등)의 일이다 — core 는 바이트를 만들 뿐 보내지 않는다.

4. 경계는 크레이트마다 다르다 — core 의 리트머스

justerm-core 를 리뷰할 땐 이 4개를 던진다. 렌더러엔 이 규칙이 안 걸린다(§5). 

① I/O 를 새로 열었나?     (파일·소켓·PTY 읽기/쓰기)   → 위반
② 뭔가를 전송하나?        (채널·IPC·네트워크로 send)  → 위반
③ 픽셀/캔버스를 그리나?    (GPU·draw·color buffer)     → 위반
④ hex/실제 색을 아는가?    (#rrggbb, 팔레트 해석)      → 위반

넷 다 "아니오"면 core 경계는 지켜진 것. 하나라도 "예"면 멈추고 되묻는다.

"theme 무지"가 왜 중요한가 core 는 색을 참조로만 저장한다: Default / Indexed(0..255) / Rgb(r,g,b). "빨강"이 화면에서 정확히 어떤 hex 인지는 절대 모른다 — 팔레트→실제색은 소비처/렌더러가 주입받아 처리. 그래서 core 에 #ff0000 이 들어오면 즉시 위반.

5. 실전 함정 둘

(1) "저스텀은 안 그린다"를 렌더러에 적용하는 실수

justerm-renderer PR 을 보고 "어? 픽셀을 그리네? 경계 위반!" — 틀렸다. 렌더러는 그리라고 만든 크레이트다. 진짜 불변식은 스코프가 있다:

core

  • 안 그린다 (§4 의 4질문)

renderer

  • 그린다 — 단 팔레트 주입(theme-agnostic) + core 침범 X

그래서 리뷰 첫 질문이 "어느 크레이트냐" 다. 크레이트를 정해야 경계가 정해진다.

(2) 우회(compensate) — 딴 층에서 몰래 메꾸기

가장 교묘한 위반. CLAUDE.md 가 "우회 금지 — 근본 층에서 고쳐라"로 못 박는다. 실사례(#297/#300):

일어난 일 core 의 이모지 폭(width) 계산에 갭이 있었다(폭은 core 책임). 그런데 렌더러가 FE0F 를 감지해 폭을 덮어버렸다. 화면은 맞게 나왔다 — 하지만 이건 우회다.
  • ① upstream(core) 버그를 가려서 고칠 압력을 없앤다.
  • ② "width 는 core 책임"이라는 경계를 침범한다.
  • ③ 같은 지식이 두 층에 중복돼 나중에 어긋난다.

올바른 처리: 렌더러 우회를 제지, 근본 수정을 core 이슈(#301)로 추적. 신호는 "테스트 통과시키려 소비처 코드로 돌아가고 있다" — 그럼 멈추고 "근본은 어느 층이냐" 물어라.

6. 왜 패밀리가 됐나 — 피벗

원래 justerm 은 "엔진만 만드는 라이브러리"였다. 그리는 건 외부 렌더러 beamterm 이 했고. 그런데 자체 렌더러 justerm-renderer 를 직접 만들며 xterm 처럼 "엔진 + 렌더링까지 다 하는 풀스택"으로 변신(pivot)했다(ADR-0018: "first-party full-stack terminal").

바뀐 것 / 안 바뀐 것 패밀리 스코프는 바뀌었다(엔진 라이브러리 → 풀스택, 렌더러가 외부→자체). 하지만 justerm-core 의 경계는 안 바뀌었다 — core 는 여전히 안 그린다. 바뀐 건 렌더러의 출신이지 core 의 경계가 아니다.
문서 읽는 법 — 방금 우리가 겪은 것 CLAUDE.md·architecture.md 는 지금도 "beamterm 이 그린다"고 쓴다. 이건 버그가 아니라 의도다 — beamterm→renderer 전환(#273)이 끝날 때까지 유지하고, 문서 플립은 #274로 tracked. 교훈: 이 repo 에서 "지금 뭘 하는가/왜 바뀌었나"의 최신 진실은 CLAUDE.md 본문이 아니라 최신 ADR(0012·0018)에 있다. 리뷰 때 "왜 이렇게 했지?"는 항상 ADR 부터.
한 줄 요약 justerm = 풀스택 패밀리(엔진+렌더러+위젯). 경계는 크레이트마다 다르다: core 는 안 그림, renderer 는 그리되 theme-agnostic. 리뷰 = "어느 크레이트냐 → 그 경계를 넘었나 → 딴 층에서 몰래 메꾸진 않나(우회)".

7. 확인 — 하나 골라 즉시 피드백

기억에서 꺼내는 게 학습이다(retrieval 연습).

Q1. "justerm 이 이제 렌더링까지 하니까 justerm-core 도 그림을 그린다." — 맞나?

Q2. 에이전트가 justerm-core 에 cell.rendered_hex = "#ff5555" 를 추가했다.

Q3. 에이전트가 justerm-renderer 에 WebGL 셰이더로 커서 사각형을 픽셀로 그리는 코드를 넣었다(팔레트는 주입받음).

Q4. core 의 커서 폭이 틀렸다. 에이전트가 렌더러에서 특정 문자를 감지해 폭을 덮어 화면을 맞췄다.

Q5. 에이전트가 justerm-core 안에서 직접 PTY 를 열어 바이트를 읽는 루프를 넣었다.
