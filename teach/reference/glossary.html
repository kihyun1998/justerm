justerm 코스 · 레퍼런스

용어집

한 번 만들면 모든 레슨이 이 정의를 따른다. 계속 자란다.

출처 정의의 authoritative 원천은 repo 의 CONTEXT.md · docs/architecture.md · CLAUDE.md. 여기 있는 건 그걸 초보 리뷰어용으로 압축한 것.
순수 터미널 엔진 pure terminal engine
바이트 → 화면 상태만 하는 층. 그리지도(렌더러 아님) 파싱만 하지도(순수 파서 아님) 않는다 — 상태 보유가 핵심. justerm 의 정체성. → 레슨 1
VT 스트림 VT byte stream
글자 + 이스케이프 시퀀스(명령)가 섞인 바이트 흐름. 셸/앱이 터미널에 뱉는 것. "VT" 는 DEC VT100 계열 터미널에서 유래.
이스케이프 시퀀스 escape sequence
\x1b(ESC)로 시작하는 명령. 색 바꾸기·커서 이동·화면 지우기 등. 예: \x1b[31m=빨강, \x1b[2J=화면 클리어. CSI/OSC/DEC 모드로 분류.
vte the crate
justerm 이 진짜 어려운 파싱만 위임하는 안정 크레이트. Paul Williams 의 ANSI 파서 상태머신 구현. 그 위의 그리드/스크롤백/selection 은 자작. → ADR-0001
셀 Cell
그리드 한 칸. 고정폭 레코드: content(grapheme), fg/bg(색 참조), attrs(bold/italic…), width(1 또는 2=CJK 전각). 빠른 디코드 위해 16바이트 고정. → architecture.md §Cell
색 참조 color reference
Default / Indexed(u8) / Rgb(u8,u8,u8). justerm 은 hex 를 영영 모른다. 팔레트→실제색은 렌더러가. = "theme 무지". → 레슨 1
grapheme cluster 셀 content
사람이 "한 글자"로 보는 단위 (결합 문자·이모지 포함). 코드포인트 ≠ 글자일 수 있어서 이 단위로 저장. 흔한 건 인라인, 드문 다중 코드포인트는 side-table 참조(셀 고정폭 유지).
damage 변경분
지난 프레임 이후 바뀐 줄 범위 + 각 줄의 바뀐 열 범위({line, left, right}). 전체 화면 다시 그리기(낭비)도, 셀 단위(과함)도 아닌 올바른 grain. 효율 축이지 품질 축 아님 — 같은 픽셀. → ADR-0003, 레슨 2
scroll op 스크롤 연산
damage 와 별개의 1급 연산: "줄을 N칸 밀고 새 줄 추가". 스크롤을 다시-그리기가 아니라 내용 이동으로 처리해 효율↑. 폭주/리사이즈 땐 전체-dirty 로 degrade.
스크롤백 scrollback
화면 위로 밀려 사라진 과거 줄들의 링 버퍼. core 가 보유 → 복사·검색이 과거까지 닿음. O(1) 스크롤은 링 구조로. → ADR-0009
뷰포트 viewport
지금 보이는 창(rows 만큼). 스크롤백 위를 미끄러지는 창문. core 가 scroll offset + follow-bottom("새 출력 오면 따라갈까 멈출까")까지 보유.
alt-screen 대체 화면
vim/less 같은 전체화면 앱이 쓰는 별도 화면(DEC 모드). 나갔다 오면 원래 화면 복귀. 엔진이 현재 화면을 투명하게 노출.
cadence ack-paced state-diff
소비처가 마지막으로 "받았다(ack)"고 한 상태 → 현재 의 diff 를 준다. 소비처가 프레임 적용 후 ack 하면 다음 diff. 느린 소비처는 중간 상태가 하나로 합쳐짐(pile-up 없음). 타이밍(언제 당길지)은 소비처 몫 — core 는 diff 만. → #13, ADR-0003
selection 선택
엔진 소유. char/word/line/block 타입. selection_text() 는 wide char·wrap 이음· 스크롤백까지 존중해 복사 텍스트 반환. 커서 blink 는 소비처 몫(애니메이션).
직렬화 / frame wire format
damage 한 사이클을 이진(고정 16바이트 셀 레코드 + grapheme side-table)으로 encode. 포맷은 core 가 주되 전송은 소비처. round-trip(encode→decode)이 테스트. → ADR-0005
core / 소비처 경계 mechanism vs policy
메커니즘이 ①VT 파싱이거나 ②버퍼 전체가 필요하면 core. 정책(query·regex·palette)은 소비처가 주입 → core 는 policy/theme-agnostic. "메커니즘 core, 정책 소비처". → ADR-0017
우회 compensate / bypass
딴 층(주로 소비처/렌더러)에서 core 결함을 몰래 메꾸는 것. 금지 — upstream 버그를 가리고 경계를 침범하고 지식을 중복시킨다. 근본 층에서 고치거나 이슈로 추적. → 레슨 1, #297/#301
소비처 consumer
justerm 을 쓰는 쪽. 첫 소비처 = PenTerm(Tauri 터미널 앱). I/O·IPC·전송·색해석은 소비처 책임.
beamterm 외부 렌더러 (전환 중)
원래 justerm 이 쓰던 외부 WebGL2 렌더러(© Adrian Papari, MIT). 단일 저자가 v1.0.0 후 정체 → 자체 렌더러로 대체 중. 지금도 CLAUDE.md·architecture.md는 "beamterm 이 그린다"고 쓰지만 그건 전환(#273) 완료까지의 의도된 옛말(#274 로 tracked). → ADR-0002, 0018
justerm-renderer 자체 렌더러
beamterm 을 대체하는 first-party WebGL2 렌더러(Rust→wasm, GL via glow). 그리는 게 임무 — 하지만 팔레트를 소비처가 주입받아 여전히 theme-agnostic. 서브셀 제어(커서 모양· undercurl·이미지)를 위해 자체 GL 층을 소유. 워크스페이스 밖(web-sys/glow=wasm 전용). → ADR-0012, 0018
피벗 / 풀스택 the pivot
justerm 이 "엔진 라이브러리" → "first-party 풀스택 터미널" 로 변신한 것(vision choice). 렌더러가 외부→자체가 되며 xterm.js/VS Code 사분면으로 이동. 단 core 경계는 불변 — justerm-core 는 여전히 안 그린다. 바뀐 건 렌더러의 출신이지 core 의 경계가 아니다. 리뷰 시 "어느 크레이트냐"를 먼저 물어야 하는 이유. → ADR-0018, 레슨 1 §7
패밀리 -term family
umbrella 이름 justerm 아래 크레이트들: justerm-core(엔진)· justerm-renderer(렌더러)·justerm-web(웹 통합)·justerm-wasm-decode· justerm-facade(0.5.1 묘비). → ADR-0010
